BUG-01 Null crash on Recipe Detail — wrong OR operator causes force-unwrap of null lists Critical Crash / Null-Safety Always
Description

In recipe.dart, three conditional guards use the || (OR) operator instead of && (AND). When recipe.ingredients, recipe.steps, or recipe.tips is null, the first condition (!= null) evaluates to false. Due to ||, Dart then evaluates the second sub-expression, which force-unwraps the same nullable field with !. This produces a Null check operator used on a null value runtime exception and crashes the app.

File References
lib/features/recipe/recipe.dart : 243
lib/features/recipe/recipe.dart : 307
lib/features/recipe/recipe.dart : 408
Reproduction Steps
  1. Open the app
  2. Navigate to the Recipe tab
  3. Tap any recipe whose JSON data has a null ingredients, steps, or tips field
  4. The Recipe Detail screen opens and immediately crashes
Observed Result

App crashes with: Null check operator used on a null value

Expected Result

Sections with null data are silently hidden; the rest of the recipe detail is displayed normally.

Root Cause

The logical operator is || (OR) when it must be && (AND). The intent is "show the section only if the list is non-null AND non-empty". With OR, when the list is null, the false left-hand side causes evaluation of the right-hand side, which crashes.

243- if (recipe.ingredients != null || recipe.ingredients!.isNotEmpty) ...[
307- if (recipe.steps != null || recipe.steps!.isNotEmpty) ...[
408- if (recipe.tips != null || recipe.tips!.isNotEmpty) ...[
Suggested Fix
243+ if (recipe.ingredients != null && recipe.ingredients!.isNotEmpty) ...[
307+ if (recipe.steps != null && recipe.steps!.isNotEmpty) ...[
408+ if (recipe.tips != null && recipe.tips!.isNotEmpty) ...[
Verification

Open a recipe with null ingredients/steps/tips. Confirm sections are hidden without crash. Open a recipe with populated lists and confirm they render correctly.

📎Appendix

Not Fully Testable

  • Runtime audio playback — just_audio / just_audio_background behavior on real device could not be verified without an emulator/device. Code path analysis was used.
  • Video thumbnail generation — video_thumbnail native plugin requires platform execution; could not verify runtime behavior of VideoThumbnail.thumbnailData().

  • Push notifications toggle — the notification checkbox in Profile stores state in memory only; actual notification permission request is not implemented (utlis/notification.dart is unused).

Observations (Not Bugs)

  • All JSON data is loaded from local assets — no network calls exist; offline behavior is not a concern for current features.
  • flutter analyze reported 0 issues — the Dart type system is satisfied by the code as written (null assertion operators are syntactically valid).
  • The MainShell caches pages in a list of size 5 while only 4 tabs exist — harmless but wasteful allocation.
  • Username is hardcoded as "Lucifer" in profile.dart — no user authentication is implemented; this is by design.
  • Several routes are defined (Login, Signup, AccountLink) but commented out — this is expected for an in-development app.
  • HomeController has an unused Timer? timer field and bool snackbarShown — dead code but not a bug.
  • Calorie formula uses a fixed activity factor of 1.2 (sedentary) — this is a design choice, not a bug.
