BUG-001
Force-unwrap of primaryFocus! crashes when no widget has focus
Critical Crash Risk Null Safety UI / Widgets
File
lib/features/widgets/custom_textfields.dart · line 133
Description
The searchTextField widget's onTapOutside callback uses a force-unwrap (!) on FocusManager.instance.primaryFocus. If no widget currently holds focus — which can happen during navigation transitions, rapid tab switching, or when the keyboard is dismissed programmatically — the expression evaluates to null and throws an unhandled Null check operator used on a null value exception, instantly crashing the app.
Reproduction Steps
  1. Navigate to the Recipe or Music tab (either screen uses searchTextField).
  2. Tap the search field so the keyboard appears.
  3. Quickly switch to another tab and tap back while the keyboard is still animating out.
  4. Tap anywhere outside the text field. The crash fires in onTapOutside.
Observed Result
App crashes with Null check operator used on a null value at custom_textfields.dart:133.
Expected Result
Keyboard/focus dismissed gracefully with no crash.
Offending Code
onTapOutside: (event) {
  // CRASH: primaryFocus can be null
  FocusManager.instance.primaryFocus!.unfocus();
},
Compare: safe version in same file (line 66)
onTapOutside: (event) {
  // SAFE — uses null-aware operator
  FocusManager.instance.primaryFocus?.unfocus();
},
Root Cause
Inconsistent null-safety usage. The companion buildTextField widget (line 66 in the same file) correctly uses the null-aware operator ?., but searchTextField uses !.
Suggested Fix
Replace primaryFocus! with primaryFocus?. on line 133, matching the pattern already established on line 66.
Verification
After fix, reproduce the navigation-race scenario above; confirm no crash occurs and keyboard is dismissed normally.
BUG-002
Unguarded null cast on navigation argument crashes Exercise screen
Critical Crash Risk Navigation Null Safety
File
lib/features/exercise/exercise.dart · line 43
Description
ExerciseState.initState() reads Get.arguments and assigns it directly to a String variable without any null check or type check: 
part = Get.arguments;
Get.arguments returns dynamic and will be null if the Exercise route is reached without providing arguments (e.g. deep-link, back-navigation quirk, or a developer mistake). Assigning null to a non-nullable String field at runtime throws a TypeError, crashing the screen before it renders. Additionally, if a caller passes a non-String argument the cast will also throw.
Reproduction Steps
  1. Navigate directly to /Exercise without supplying arguments (e.g. via a test harness or future refactor that forgets to pass the argument).
  2. The initState assignment crashes immediately.
Observed Result
TypeError: Null is not a subtype of type 'String' in initState.
Expected Result
Graceful fallback (e.g. empty string or redirect back to shell) if the argument is absent.
Offending Code
void initState() {
  super.initState();
  part = Get.arguments;  // dynamic → String without guard
  exerciseController.loadMuscleData(part);
  exerciseController.loadExerciseData(part);
}
Root Cause
No null-check or type-cast guard around Get.arguments. The Dart analyser does not flag this because Get.arguments is typed dynamic, which is assignment-compatible with any type.
Suggested Fix
Use a guarded cast: part = Get.arguments as String? ?? ''; and handle the empty case (e.g. pop back to shell).
BUG-003
Route argument key mismatch: MainShell binding always falls back to Home (index 0)
High Navigation Logic Error State Management
Files
lib/features/main_shell/binding/binding.dart · line 9 lib/routes/routes.dart · line 31
Description
The MainShellBinding reads the navigation argument with key 'index', but every caller that navigates to the shell route passes the key 'idx'. The map lookup always returns null, so the ?? 0 fallback is always used — the binding always initialises the Home controller regardless of which tab was requested.
Offending Code — binding
// MainShellBinding reads 'index'
final int index = Get.arguments?['index'] ?? 0;
Callers use key 'idx'
// routes.dart line 31
page: () => MainShell(initialIdx: Get.arguments?['idx'] ?? 0),

// exercise.dart line 261, etc.
Get.offAllNamed(MyRoutes.shellRoute, arguments: {'idx': 0});
Impact
When a user navigates back to the shell from Exercise or Recipe pages, the shell correctly shows the right tab via initialIdx in the page widget, but the binding silently skips registering the correct controller. If the controller for the displayed tab was already registered (e.g. from a prior visit) this is invisible. But on first cold-start deep-link or if a controller has been disposed, the shell may render a tab whose controller is not registered, causing a GetxController not found exception.
Root Cause
Binding and page use different argument key strings ('index' vs 'idx'). A rename was applied to one location but not the other.
Suggested Fix
Standardise on one key, e.g. 'idx', and update MainShellBinding line 9 to Get.arguments?['idx'] ?? 0.
BUG-004
ExerciseController race condition silently discards all exercise data
High Race Condition Async State Management
File
lib/features/exercise/controller/controller.dart · lines 11–33 lib/features/exercise/exercise.dart · lines 44–45
Description
loadMuscleData and loadExerciseData are launched as two separate async operations that each independently set isLoading.value = true at the start and isLoading.value = false in their finally block. Both functions run concurrently. The first one to finish will set isLoading.value = false, causing the UI to leave the loading state and attempt to render before the second operation completes. Because the exercise data load is significantly heavier than the muscle data load, the UI typically renders with an empty exercise list. There is also a silent discard: if loadMuscleData finishes first it sets isLoading = false, then loadExerciseData subsequently calls isLoading = true again — causing a loading spinner flash — before setting it to false a second time. Any filter UI re-render during this window sees stale data.
Offending Code
void initState() {
  super.initState();
  part = Get.arguments;
  exerciseController.loadMuscleData(part);   // launched, not awaited
  exerciseController.loadExerciseData(part); // launched, not awaited
}
// Inside controller — each manages isLoading independently:
Future<void> loadMuscleData(String part) async {
  isLoading.value = true;
  // ...
  finally { isLoading.value = false; }  // can fire before loadExerciseData finishes
}
Future<void> loadExerciseData(String part) async {
  isLoading.value = true;
  // ...
  finally { isLoading.value = false; }
}
Root Cause
Two independent async methods share a single isLoading boolean without coordination. Neither Future.wait nor a loading counter is used.
Suggested Fix
Introduce a single coordinating method that uses Future.wait([ loadMuscleData(part), loadExerciseData(part) ]) and manages isLoading around the combined future — similar to the pattern already used in HomeController.loadHomeData() and MusicController.loadMusicData().
BUG-005
Wrong route binding for CalorieCalculator causes controller registration conflict
High Navigation Dependency Injection GetX
File
lib/routes/routes.dart · lines 50–54
Description
The calorieCalculatorRoute GetPage uses RecipeBinding() as its binding. This is correct in the sense that CalorieCalculator uses RecipeController. However, the route definition creates a duplicate path alongside the recipeListRoute, which also uses RecipeBinding(). When navigating to the Calorie Calculator via its named route (/CalorieCalculator), GetX attempts to call RecipeBinding().dependencies(), which calls Get.lazyPut(() => RecipeController()). If a RecipeController is already registered (from the Recipe tab), GetX will silently use the cached instance — but there is no fenix: true or permanent flag, so if the instance was disposed via a previous Get.offAllNamed stack pop, the re-registration fails silently or returns a disposed controller. The CalorieCalculator then calls Get.find<RecipeController>() on a disposed object, causing silent data loss or runtime exceptions.
Offending Code
GetPage(
  name: calorieCalculatorRoute,
  page: () => const CalorieCalculator(),
  binding: RecipeBinding(),  // same binding as recipeListRoute
),
Root Cause
The CalorieCalculator route is only ever used inside MainShell (tab index 3), not as a standalone named route. The named route registration is therefore redundant but also risky because it lacks proper controller lifecycle management for its dual-usage context.
Suggested Fix
Either remove the named route and rely only on the MainShell tab mechanism, or add fenix: true to the Get.lazyPut call in RecipeBinding so that the controller can be safely re-created after disposal.
BUG-006
Get.offAllNamed(exerciseRoute) pops all routes — Music keeps playing but Music tab is destroyed
High Navigation State Bug Audio
File
lib/features/home/home.dart · line 179
Description
When the user taps a workout card on the Home tab, the code calls: 
Get.offAllNamed(MyRoutes.exerciseRoute, arguments: part);
offAllNamed clears the entire navigation stack and replaces it with the Exercise screen. This means the MainShell widget is disposed. If the user had music playing via the Music tab (which persists via permanent: true on MusicController), the audio continues in the background (which may be desirable), but the mini-player UI in MusicState is destroyed. If the user navigates back to the Music tab later, a new MusicState is created that still references the alive MusicController, which is fine — but the expandedIdx and assetFileCache local state of the old MusicState are lost, creating a confusing UI where the previously expanded playlist is collapsed.

More critically: every other tab's cached page widget is discarded by offAllNamed. The search field text in RecipeList and the selected filter state in Exercise are silently cleared on return.
Root Cause
Get.offAllNamed is too aggressive — it should be Get.toNamed (push without clearing stack) or the shell architecture should be maintained throughout.
Suggested Fix
Use Get.toNamed(MyRoutes.exerciseRoute, arguments: part) to preserve the MainShell in the stack, and update the Exercise screen's back navigation to use Get.back() instead of Get.offAllNamed(shellRoute).
BUG-007
Macro scoring in CalorieCalculator multiplies macros by serve but recipe JSON stores per-serve values
High Logic Error Calculation CalorieCalculator
File
lib/features/calorie_calculator/calorie_calculator.dart · lines 141–145
Description
When scoring recipes against the user's macro targets, the code multiplies each recipe's nutritional values by serve: 
final serve = (r.serve ?? 1).toDouble();
final calTotal    = (r.cal    ?? 0) * serve;
final proteinTotal = (r.protien ?? 0) * serve;
This implies the JSON stores nutrition values per single unit, and the total is obtained by multiplying by the number of servings. However, the Recipe detail screen (recipe.dart) displays recipe.cal directly as the total — it does not multiply by serve. This confirms the JSON stores the total nutrition for the recipe (not per-serving). Multiplying by serve in the calculator therefore overstates total macros (e.g. a recipe with 500 cal and 2 servings is scored as 1000 cal), causing most recipes to fail the threshold check and producing incorrect "No recipes match" results even when appropriate recipes exist.
Root Cause
Misinterpretation of the data model: serve is the number of servings, but cal is the total recipe calorie count, not per-serving. The scoring code incorrectly treats cal as per-serving.
Suggested Fix
Remove the multiplication by serve and compare recipe totals directly against the user's targets. If per-serving comparison is intended, the recipe card display and the scoring logic must be made consistent.
BUG-008
Recipe detail screen accessed with id=0 when recipe card's id is null
High Null Safety Navigation Data Error
File
lib/features/widgets/custom_card.dart · line 14 lib/features/recipe/service/service.dart · line 27
Description
The recipe card navigates to the detail screen using: 
Get.to(() => Recipe(id: recipe.id ?? 0));
If recipe.id is null (which can happen if the JSON entry omits the id field), the fallback 0 is passed. The service then calls data.firstWhere((r) => r.id == 0). If no recipe has id 0 (which is likely since IDs typically start at 1), firstWhere throws a StateError: No element, crashing the recipe detail screen.
Root Cause
Using a sentinel value of 0 for a missing id, combined with no error handling in the service for the case where firstWhere finds no matching recipe.
Suggested Fix
Guard the navigation: if (recipe.id == null) return; before navigating. In the service, use firstWhereOrNull and handle the null case in the controller.
BUG-009
RecipeModel initialised with all-null fields; detail screen renders null values before load
Medium State Bug UI Recipe
File
lib/features/recipe/controller/controller.dart · line 9 lib/features/recipe/recipe.dart · lines 63–65, 139, 177, 204, 212, 220, 228
Description
The controller initialises recipe as Rx<RecipeModel>(RecipeModel()). A freshly constructed RecipeModel() has all fields null. The Recipe widget loads the recipe via a postFrameCallback, meaning there is a guaranteed one-frame window (and potentially multiple frames if the device is slow) where the Obx widget renders with isLoading = false but recipe containing all nulls.

During that window, the widget renders '${recipe.heading}' as the string "null", and recipe.image ?? BUG-011
Video playback: stopPlayback called asynchronously inside Obx rebuild triggers a frame-callback warning
Medium Async UI Bug Exercise
File
lib/features/exercise/exercise.dart · lines 323–326
Description
Inside the Obx rebuild callback (which is called synchronously during the build phase), stopPlayback(refreshUi: false) is called directly when playingIndex is out of range after a filter change:

Calling async I/O operations during a widget build is not allowed and causes Flutter to emit build-phase side-effect warnings. BUG-013
int.parse('0xFF${exercise.bgColor}') throws on null or invalid hex color string
Medium Crash Risk Data Parsing Home
File
lib/features/home/home.dart · line 183 lib/features/music/music.dart · line 445
Description
Both home.dart and music.dart construct Color objects by parsing a hex string from the JSON model: 
Color(int.parse('0xFF${exercise.bgColor}'))
Color(int.parse('0xFF$color'))   // music.dart
int.parse throws a FormatException if:
  • bgColor / color is null (produces '0xFFnull')
  • The value contains characters outside [0-9A-Fa-f]
  • The value is the wrong length
Since bgColor and color are nullable String? in the models, a missing JSON key causes a FormatException and crashes the entire list render.
Root Cause
No null-check or try-catch guard around int.parse for user-supplied color strings.
Suggested Fix
Use int.tryParse('0xFF${exercise.bgColor ?? 'D9D9D9'}') ?? 0xFFD9D9D9 to provide a safe fallback color.
